## Lessons Learned
- **WisprFlow prompts are 100% server-side.** The client is a thin sender of audio + context (app name, textbox contents, dictionary, AX/OCR context). No system prompts extractable from the binary. Only the Polish feature instruction labels are local ("Make more concise", "Reword for clarity", etc.).
- **Granola prompts are also server-side** for core summarization (`meeting-summary-consolidated`, etc.), but **recipe templates are cached locally** in `~/Library/Application Support/Granola/cache-v4.json` — this is where the 47 full prompt instructions came from.
- **Granola uses prompt slugs** (`prompt_slug` + `prompt_variables`) — client sends slug name and variables, server resolves to actual template. Slugs include: `notes-centric-long-input`, `meeting-summary-consolidated`, `template-summary-consolidated`, `did-meeting-end`, `meeting-title`, etc.
- **Granola LLM output format**: XML-like `<title>...</title>` and `<notes>...</notes>` tags parsed client-side.
- **WisprFlow uses OpenPipe fine-tuned Llama models** for English formatting: `openpipe:llama3-1-8b-12-09-2024` (general) and `openpipe:llama3-1-05-21-data-email` (email-specific).
- **Granola dynamic heading count** based on transcript length: <8K chars → 4 headings, <20K → 5, <80K → `floor(len/10000)+1`, else 8.

## Nuances & Edge Cases
- WisprFlow sends extensive context to server: current app type (email/chat/IDE), textbox before/after cursor, selected text, VS Code variable/file names, user dictionary with replacements and snippets, personalization style per context (formal for work/email, casual for personal).
- Granola's 30 panel templates define meeting-type-specific section structures (e.g., "VC: Pitch" → Team/Problem/Product/Traction/Funding sections; "Sprint Planning" → Goals/Assignments/Dependencies/Risks). These could inform Muesli's template system.
- Granola recipe "coach-me-Matt" is 14,448 chars — the Mochary Method coaching curriculum embedded as a prompt. Shows the scale of prompt engineering in production meeting apps.
